# LongInvest AI 开发守则

## 1. 目的与适用范围

本守则用于约束 LongInvest 项目中的 AI 开发工作，在不降低关键业务可靠性的前提下，减少重复规划、无效验证、过度设计和反复全量验收。

本守则适用于需求分析、设计、编码、测试、迁移、集成、部署和问题修复。需求与技术行为仍以 `docs/requirements/README.md` 指向的唯一生效规格为最高依据；本守则规定如何高效、可靠地完成这些需求。

规则冲突时按以下顺序处理：

1. 当前生效的详细需求规格。
2. 根目录 `AGENTS.md`。
3. 本守则。
4. 已批准的当前批次设计与实施计划。
5. 代码中的一般惯例。

## 2. 核心原则

1. **规格优先**：开工前读取当前任务直接对应的详细规格，不重复解读已经明确的内容。
2. **直接推进**：除需求歧义、规格冲突、不可逆操作、用户改动冲突或公共契约变更外，AI 应自行作出实现判断并推进到完成。
3. **风险匹配**：验证范围必须与改动风险相匹配，禁止无条件运行所有测试。
4. **整批验收**：同一开发批次只在集成完成后执行一次全量验收、一次必要的容器验收和一次部署验收。
5. **最小改动**：不扩大需求，不重构无关代码，不升级无关依赖，不提前建设未来能力。
6. **证据完成**：代码写完不是完成；必须实际运行与风险相匹配的检查并查看结果。
7. **失败闭环**：验证失败后先定位原因，修复并复验，不把可自行解决的半成品交给用户。
8. **及时停止**：达到需求和验收条件后立即结束，不用额外文档、抽象或测试延长任务。

## 3. 风险分级

每个任务开工时必须确定风险等级。命中任一更高等级条件时自动升级；AI 不得为了减少验证而自行降级。

### 3.1 L1 低风险

适用范围：

- 文案、注释、开发文档和测试说明。
- 小范围样式或展示调整。
- 模块内部纯函数或局部错误修复。
- 不改变公开接口、数据库、权限和业务状态的调整。

最低验证：

- 运行直接相关的测试或进行代表性实际检查。
- 对涉及文件运行适用的格式、静态或类型检查。
- 页面改动检查目标页面；涉及响应式布局时检查桌面和移动端。

禁止为 L1 任务运行无关的全量后端、全量前端、迁移或容器验收。

### 3.2 L2 中风险

适用范围：

- 新增模块内部功能或普通接口。
- 修改 Service、Repository、Worker 或前后端交互。
- 修改本模块拥有的数据表。
- 修改任务处理、失败重试或普通状态流转。
- 改动影响一个或两个业务模块，但不改变公共基础设施。

最低验证：

- 验证主要正常路径和最可能发生的异常路径。
- 运行本模块全部测试和适用的静态检查。
- 根据影响运行相关集成测试。
- 存在迁移时验证升级和约束结果。

L2 子任务完成后不得重复执行整仓全量检查；全量检查留到批次集成门槛。

### 3.3 L3 高风险

适用范围：

- 公共契约、认证、授权、安全和密钥处理。
- 价格、金额、目标、信号、持仓、通知资格等关键业务规则。
- 幂等、并发、事务、状态机、事务发件箱和恢复机制。
- Alembic 主链、跨模块迁移和历史数据兼容。
- 公共入口、任务框架、Compose、依赖、部署和生成接口类型。
- 同时影响三个或更多模块的改动。

最低验证：

- 覆盖正常、异常、重复提交、并发和恢复场景中与改动相关的部分。
- 运行相关模块测试与集成测试。
- 在整批集成后运行一次受影响端的全量检查。
- 涉及数据库运行环境、进程编排或部署时执行一次容器验收。

## 4. 开发流程

### 4.1 开工

每个开发批次按以下顺序开始：

1. 读取 `docs/requirements/README.md` 和当前任务对应的详细规格章节。
2. 检查工作区状态，识别并保护已有未提交修改。
3. 明确本批目标、不做内容、风险等级和完成条件。
4. 列出涉及模块、模块边界、公共文件和依赖顺序。
5. 选择与风险匹配的最小验证集合。

如果详细规格已经明确，不得重新发起开放式需求讨论。只有不同选择会明显改变产品行为时才询问用户。

### 4.2 规划

规划必须足以指导施工，但不得复制规格或预写整份实现。计划只需包含：

- 目标和不做内容。
- 对应规格章节。
- 风险等级及原因。
- 涉及模块、公共文件和数据迁移。
- 模块拥有的数据、公开接口、事件和错误码。
- 任务依赖、可并行项和串行项。
- 每项完成条件和本批最终验收命令。

规划深度按风险调整：

- L1：可直接施工，不强制创建独立计划。
- L2：使用简要计划，不粘贴大段预期代码。
- L3：使用完整设计和批次计划，重点写清契约、事务、并发、迁移与恢复。

禁止：

- 把写测试、运行失败、写实现、运行成功和提交机械拆成大量微步骤。
- 在计划中重复抄写规格原文。
- 为每个小任务单独创建设计文档。
- 把提交次数或文档长度当作完成标准。
- 已经明确的内部实现反复请求用户批准。

### 4.3 实施

1. 优先完成模块内部实现和模块测试。
2. 子模块不得自行修改公共入口、公共契约、迁移主链、Compose、依赖锁文件和生成类型。
3. 公共文件由主流程在模块稳定后串行接入。
4. 每完成一个模块只运行模块级验证。
5. 整批集成完成后再运行一次全量验收。
6. 验收失败时先运行最小失败用例，修复后只复验受影响范围；只有公共行为再次变化时才重跑全量验收。

### 4.4 完成

任务只有同时满足以下条件才能报告完成：

- 实现符合生效规格和当前批准范围。
- 模块边界与公开契约没有被破坏。
- 对应风险的验证已经实际通过。
- 没有调试代码、占位实现、静默失败或本次引入的无用代码。
- 迁移、生成文件和运行配置与代码保持一致。
- 工作区中的用户原有修改得到保留。

## 5. 架构规范

### 5.1 总体架构

项目保持模块化单体：一个代码仓库、一个主要数据库，API、调度器和不同 Worker 可独立运行。PostgreSQL 是业务与任务状态的真实来源；Redis 和 RQ 只负责任务传递和临时协调。

现有架构能够满足需求时，禁止提前拆分微服务或引入服务注册、分布式事务、新消息中间件等额外基础设施。

### 5.2 模块所有权

每个业务模块必须明确：

- 业务职责和明确不负责的内容。
- 拥有的数据表。
- 公开 Service 和 HTTP 接口。
- 发布、接收的事件和任务。
- 稳定错误码。
- 外部依赖和验收场景。

模块只能直接读写自己拥有的表。禁止导入其他模块的数据库模型、调用其他模块的 Repository、直接修改其他模块数据或依赖其内部函数。

跨模块读取使用公开查询 Service 或只读集成契约；跨模块写入使用公开命令 Service、事务发件箱或明确任务入口。

### 5.3 模块内部职责

后端模块按实际需要划分以下职责，不要求为空职责机械创建文件：

- API：认证授权、输入转换和响应转换。
- Schema：公开输入输出结构。
- Service：业务用例、权限后的业务协调和事务边界。
- Domain：业务不变量、状态判断和计算规则。
- Repository：本模块数据库读写。
- Model：本模块数据库模型。
- Event：领域事件和集成事件。
- Error：稳定业务错误码。
- Job：异步任务入口。

依赖方向必须保持为 API 或 Worker 调用 Service，Service 调用 Domain 和 Repository。Repository 不得调用 Service；Domain 不得依赖 FastAPI、SQLAlchemy Session、RQ 或具体第三方客户端。

### 5.4 公开契约

HTTP 接口、公开 Service、模块间数据对象、事件、任务载荷、错误码、OpenAPI 和生成的前端类型均属于公开契约。

修改公共契约时必须说明调用方影响、兼容性、迁移方式、旧任务或旧事件处理，以及是否需要重新生成前端类型。禁止用无结构字典、任意字段或类型逃逸绕过正式契约。

### 5.5 数据设计

- 每张表必须有唯一明确的所属模块。
- 主键、唯一约束、外键和检查约束承担真实数据不变量。
- 幂等要求必要时必须由数据库唯一约束兜底。
- 价格、金额和比例计算使用精确数值，禁止使用浮点数保存金额和价格。
- 时间统一保存为带时区时间，业务时间必须明确使用的时区。
- 状态使用受控枚举或明确约束。
- 需要审计、恢复或复现的数据保存不可变历史。
- JSON 只用于结构确实多变的数据，不得代替正常关系设计。
- 索引服务于实际查询、唯一性和锁定路径，不为假设需求提前堆积索引。

### 5.6 事务与并发

一次业务操作中要求共同成功或共同失败的数据必须在同一事务中完成，例如当前状态与历史、业务事实与审计、数据变更与发件箱事件、当前指针与不可变版本。

事务内禁止执行长时间外部网络请求。跨进程衔接使用事务发件箱，禁止先提交数据库后直接发送消息。

并发写入必须明确采用唯一约束、乐观锁、行锁、条件更新、幂等键或确定性任务键中的适当机制，不能依赖“通常不会同时执行”。

### 5.7 异步任务

异步任务必须具备稳定任务类型、明确输入结构、有限超时、有限重试、幂等处理、可查询状态、失败原因、失败隔离和中断恢复方式。

任务载荷只保存标识符和必要快照，不传递数据库对象。批量任务按单项记录结果；单项失败不得回滚其他已成功项目，除非规格明确要求整批原子提交。

### 5.8 外部依赖

第三方数据源通过适配器隔离。字段转换、超时、重试、限流、熔断和错误转换在边界集中处理，第三方原始响应不得扩散到业务层。

普通自动测试使用可控替身，不依赖实时网络；真实外部服务只在专门联调或验收阶段调用。

### 5.9 API 架构

API 层不得编写复杂业务规则、拼接数据库查询、直接提交业务事务或调用其他模块内部代码。

写接口必须按规格处理身份、权限、CSRF、输入校验、幂等、状态冲突、审计和安全错误输出。列表接口必须明确分页、排序和过滤，禁止无上限返回大表。

### 5.10 前端架构

前端按业务功能组织页面、组件、请求、表单和测试。共享目录只保存真正被多个功能复用的内容。

服务端状态由请求缓存层管理，页面临时状态保留在组件内，表单状态由表单工具管理；只有确有跨页面长期共享需求时才使用全局状态。禁止把同一服务端数据复制到多个状态容器并手工同步。

## 6. 编码规范

### 6.1 简单与边界

- 选择满足当前规格的最简单方案，优先现有模式、标准库和已安装依赖。
- 不为单一调用创建通用框架，不提前实现后续阶段能力。
- 只有抽象能消除真实重复或明确复杂度时才引入。
- 只修改完成当前目标所需文件，不整理无关格式、不升级无关依赖、不改变无关公开名称。
- 发现无关问题可以记录，但不得纳入当前施工。

### 6.2 命名

- 名称表达业务含义；类名使用名词，函数名使用动作，集合使用复数。
- 布尔值使用 `is`、`has`、`can`、`should` 等明确前缀。
- 多对象上下文中的标识符必须说明对象，避免含义不清的通用 `id`。
- 避免无意义的 `data`、`info`、`obj`、`temp`、`handle`。
- 业务枚举、事件名、任务名和错误码必须稳定，不得随意更换同义词。

### 6.3 函数与控制流

- 一个函数只完成一个可以清楚描述的工作。
- 输入、输出和失败方式必须明确，减少隐藏状态和调用顺序依赖。
- 业务判断与数据库访问分离。
- 优先使用提前返回降低嵌套层级。
- Service 负责协调读取、规则判断、写入和事件；独立规则放入可单独测试的领域函数。

### 6.4 类型

Python 公开函数和业务核心函数必须声明参数及返回类型。API Schema、数据库模型和模块间契约不得混用同一个对象；稳定业务结构不得用无结构字典替代。

TypeScript 禁止无理由使用 `any` 或强制断言掩盖不确定数据。外部输入必须校验，接口响应优先使用 OpenAPI 生成类型；可空、未加载和不存在必须分别表达。

### 6.5 错误处理

- Domain 产生业务错误，Service 补充业务上下文，API 转换为标准响应，Worker 转换为任务失败或部分失败，适配器转换外部错误。
- 禁止捕获所有异常后忽略、用 `None` 表示多类失败、依赖错误消息文本判断逻辑。
- 禁止把所有失败返回成功或统一转为同一种错误。
- 永久性业务错误不得自动重试。
- 不向用户暴露 SQL、堆栈、服务器路径、密钥和第三方完整响应。

### 6.6 日志

日志记录请求编号、任务编号、业务对象编号、操作、结果、耗时、重试次数和安全处理后的错误类别。

禁止记录密码、Session、Token、密钥、完整敏感信息和第三方完整响应；禁止无意义的进入/离开函数日志。同一异常通常只在最终处理边界记录一次。

### 6.7 注释与文档

注释用于说明不直观规则、特殊边界、锁和事务原因、第三方限制，不重复代码动作，不保存删除代码，不以长篇注释弥补混乱结构。

公开契约、复杂状态机和迁移兼容规则需要文档；普通内部函数不机械添加说明。

### 6.8 数据访问

- Repository 负责查询和持久化，不负责业务决策。
- 查询必须明确结果数量，批量读取避免逐条查询。
- 更新操作检查实际影响行数，分页必须使用稳定排序。
- 事务由 Service 或工作单元控制，Repository 内不得偷偷提交。
- 禁止为了测试方便削弱生产数据库约束。

### 6.9 后端工具约束

- Python 版本和依赖以 `backend/pyproject.toml` 为准。
- Python 代码遵守 Ruff 当前配置，行长不超过 88。
- 新代码不得通过禁用检查或扩大忽略范围掩盖问题，除非有明确、局部且可解释的原因。

### 6.10 前端编码

- 页面负责组合和路由级行为，表单负责输入和提交，展示组件尽量不包含请求逻辑。
- 数据转换集中在请求边界或适配函数。
- 异步操作必须处理加载、成功、空数据、失败和必要的重试状态。
- 删除、覆盖、发布等重要操作必须确认。
- 重复提交必须通过禁用、防抖或服务端幂等保护。
- 错误信息应说明发生了什么以及用户可以采取的行动。
- 禁止刷新整个页面来掩盖状态同步问题。
- 生成的接口类型不得手工修改。

## 7. 测试规范

### 7.1 选择原则

所有改动至少验证主要正常路径、本次最可能破坏的路径，以及已修复问题的回归场景。

仅在存在对应风险时增加空数据、非法输入、权限、重复提交、并发、超时、外部依赖失败、部分失败隔离、中断恢复、迁移升级和旧数据兼容测试。

### 7.2 测试质量

- 一个测试验证一个清晰、可观察行为。
- 测试名称说明场景和预期结果。
- 回归测试必须能够在修复前暴露问题。
- 时间、随机数和外部请求必须可控。
- 测试不得依赖执行顺序。
- 关键事务行为不能只用模拟代替真实数据库验证。
- 不测试框架已经保证的行为，不为简单赋值机械增加测试。
- 不用测试数量、覆盖率数字或快照数量代替业务风险判断。

### 7.3 禁止的无效验证

- 纯展示修改测试超时或数据库迁移。
- 后端局部函数修改运行无关前端全量构建。
- 未修改容器或运行配置时反复创建全新容器环境。
- 同一阶段反复运行完全相同的全量检查。
- 只为增加测试数量编写没有业务价值的用例。
- 失败后不分析原因而盲目重跑整套检查。

## 8. 验收规范

### 8.1 局部门槛

- 直接相关测试通过。
- 适用的静态、格式或类型检查通过。
- 实际输出符合当前需求。

### 8.2 模块门槛

- 模块测试通过。
- 相关集成契约通过。
- 与风险对应的主要异常路径通过。
- 没有越过模块所有权边界。

### 8.3 批次门槛

- 所有模块已接入，公共文件已串行修改完成。
- 后端全量测试、Ruff 和必要编译检查只执行一次。
- 前端全量测试、ESLint、类型检查和生产构建只执行一次。
- OpenAPI 与生成类型一致。
- Alembic 保持预期的单一迁移主链。
- 只有涉及运行环境、迁移或进程编排时才执行一次全新容器验收。

### 8.4 发布门槛

- 部署目标对应已通过验收的提交。
- 数据库迁移成功，健康检查和关键入口正常。
- 后台进程没有异常重启，不引入计划外公开端口。
- 发布后只进行关键冒烟检查，不在服务器重复开发期全量测试。

## 9. 并行与公共文件

公共底座通过验收后，每批最多并行三个互不依赖的模块，不为凑数强行并行。

可以并行：

- 不互相依赖的业务模块。
- 不修改相同文件的前端功能。
- 各模块自己的实现和测试。
- 独立的资料调查和问题诊断。

必须由主流程串行维护：

- 应用入口和主路由。
- 公共契约。
- Alembic 环境和迁移主链。
- Compose 和部署文件。
- 依赖与锁文件。
- OpenAPI 导出和前端类型生成。
- 跨模块集成测试。

并行子任务发现需要修改公共内容时必须停止该部分施工，说明影响和所需变更，由主流程统一处理。

## 10. 失败处理

验证失败后先分类，再决定复验范围：

- **本次改动导致**：修复后重跑失败项及其直接上游。
- **仓库原有问题**：保留证据，不扩大当前任务；必须确认本次改动没有使其恶化。
- **环境问题**：修复环境或采用能证明同一行为的等价验证方式。
- **偶发问题**：最多原样重试一次；再次失败必须分析原因。
- **全量检查失败**：先缩小到最小失败用例，禁止盲目重复全量检查。

只有修复再次影响公共行为、迁移、契约或批次集成时，才重新执行整批全量验收。

## 11. AI 行为约束

AI 必须：

- 开工前定义本次完成条件和风险等级。
- 使用仓库真实代码、配置、日志和测试作为判断依据。
- 保护用户未提交修改，不执行恢复或覆盖操作。
- 在权限范围内持续完成实现、验证、修复和复验。
- 最终用清晰中文说明完成了什么、验证了什么、结果如何。

AI 只有在以下情况才应暂停并请求用户决定：

- 生效规格存在会改变产品行为的实质冲突。
- 候选方案需要修改当前生效规格或公共契约。
- 需要删除、覆盖用户已有修改或执行不可逆数据操作。
- 缺少的业务选择无法从规格和仓库中确定，且不同选择影响明显。

以下情况不得作为暂停理由：

- 普通内部命名或文件拆分选择。
- 能从现有代码模式推断的实现细节。
- 可以自行修复的测试、格式或类型错误。
- 与当前任务无关的仓库既有问题。

## 12. 停止条件

满足以下条件后立即结束当前开发，不继续增加工作：

- 用户要求和计划完成条件已经达到。
- 与风险等级相匹配的验证已经通过。
- 剩余问题与本次改动无关。
- 继续工作只会增加文档、测试、抽象或重构数量，而不会实质降低风险。
