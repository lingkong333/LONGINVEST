# A股长波段价格提醒系统 MVP 历史设计与环境补充

## 1. 文档状态

- 状态：历史实施与环境补充说明，不再是需求基线。
- 当前唯一需求基线：以 `docs/requirements/README.md` 的指向为准。
- 历史原始需求：`A股个股长波段价格提醒系统_需求与技术设计_v2.1_MVP_完整版.md`。
- 使用边界：只保留服务器资源、部署限制和施工组织等补充信息；任何冲突均以当前需求基线为准。
- 目标环境：Linux + Docker。
- 项目形态：全新独立仓库，不复用或改造服务器上的 SignalForge。
- 开发组织：公共底座由主流程完成；底座冻结后，业务模块按依赖层每批最多三个并行。

## 2. 目标与范围

系统面向单个管理员，完成以下闭环：

```text
交易日历和股票主数据
-> 全市场不复权日线与监控股前复权数据
-> 手工或策略目标
-> 盘中批量行情
-> 五区间信号状态机
-> 网页、企业微信和邮件通知
-> 单股、监控列表和全市场独立回测
```

系统不连接券商，不自动下单，不保存真实持仓数量、成本和盈亏，不提供组合回测。

## 3. 已确认的关键决策

1. 使用模块化单体，不拆微服务。
2. PostgreSQL 是业务状态和任务状态的唯一正式来源。
3. Redis/RQ 只负责任务传输、临时协调和限速；Redis 中断不能导致任务丢失。
4. 后端使用同一镜像，通过不同命令运行 API、调度、分发、看门狗和各类 Worker。
5. 策略代码在专用受限环境执行；API 和普通 Worker 不获得 Docker Socket 或宿主机控制权限。
6. 前端先封装基础组件，再用模拟数据完成页面，最后连接稳定接口。
7. 每个业务模块拥有自己的规则、表、接口、后台任务和测试，其他模块不能直接修改其数据。
8. 延迟动作和外部副作用通过 PostgreSQL 事务发件箱衔接。
9. 当前服务器可顺序回填十年至二十年左右的全市场日线，并执行单股回测；全市场并行回测需要后续扩容。
10. MVP 开发和验收不直接开放公网。公网正式使用前必须单独完成 Nginx、TLS、访问控制和恢复方案。

## 4. 服务器基线

已只读确认的目标服务器：

- Ubuntu 24.04 LTS，x86_64。
- 2 核 CPU，约 3.6GB 内存，约 36GB 可用磁盘。
- Docker、Docker Compose、Git 和 Python 可用。
- 主机 80/443 已由 Nginx 使用。
- 服务器已有其他容器，新项目不得复用其数据库、Redis、数据卷或端口。

当前阶段约束：

- 新项目使用独立 Compose 项目名、网络、数据库、Redis 和数据卷。
- 数据库和 Redis 只在容器内部网络开放，不绑定公网地址。
- 历史回填并发为 1 至 2；单股回测并发为 1。
- 日线任务优先于回测任务。
- 剩余磁盘不足 15GB 时暂停历史回填并产生告警。
- 前端在容器内构建，不依赖主机 Node 环境。

```text
LongInvest/

      entrypoints/          API、调度、分发、看门狗、Worker、CLI

      integration_contracts/跨模块公开数据与事件契约

      modules/              业务模块
    tests/
  frontend/
    src/

      api/generated/        后端接口自动生成类型
      ui/                   真正通用的基础组件
      lib/                  时间、错误和无业务含义的工具

      tests/
  deploy/
    compose.yaml
    docker/
  docs/
```

小模块不机械套用多层目录。模块变大后，内部按以下职责拆分：

```text
public       唯一允许其他模块使用的公开接口
domain       状态、规则和纯计算
service      本模块业务流程和事务
repository   本模块数据访问
models       本模块拥有的数据库表
api          HTTP 接口
handlers     后台任务和事件处理
```

## 6. 后端模块边界

### 6.1 公共底座

公共底座先完成并冻结：

- 配置、数据库连接、事务、迁移、UTC 时间、上海交易时间和 Decimal 金额规则。
- 标准响应、稳定错误码、请求 ID、幂等键和版本冲突。
- Session、CSRF、管理员 CLI 和登录限速。
- `job/job_run/job_item`、事务发件箱、分发、心跳、围栏令牌、超时、取消和恢复。
- 结构化日志、只追加审计、健康检查、基础系统告警。
- 密钥加密、掩码、配置历史和任务配置快照。
- HTTP 超时、有限重试、限速、熔断和响应校验。
- 通知渠道、模板、投递、尝试、重试和测试发送的通用能力。

### 6.2 业务模块及数据所有权

| 模块 | 独占职责和数据 |
|---|---|
| calendar | 交易日、交易时段和日历版本 |
| securities | 股票身份、市场、类型、上市状态和代码映射 |
| providers | 外部数据适配、能力路由、限速、熔断和诊断 |
| scheduling | 调度定义、版本和执行记录，不执行行情 |
| market_data | 行情批次、日线、前复权、数据质量和数据修订 |
| monitoring | 分组、订阅和订阅版本 |
| positions | 当前持仓标记和不可变持仓历史 |
| strategies | 草稿、版本、验证、发布资料和受限执行 |
| targets | 目标版本、当前绑定、计算记录和人工复核 |
| signals | 当前区间、判断记录和信号事件 |
| notifications | 通知策略、事件、渠道投递和投递尝试 |
| backtests | 回测任务、订单、交易、指标和单股结果 |
| operations | 系统告警、健康详情和仪表盘只读汇总 |

### 6.3 禁止事项

- 模块不得引用其他模块的 ORM 模型、Repository 或内部文件。
- 不建立通用业务 Repository、万能 Service 或不断扩张的 `utils.py`。
- 一个模块不得直接更新或删除另一个模块的表。
- 数据库不得通过级联删除跨模块历史。
- 任务、配置、日志和审计不得承载业务判断。
- 仪表盘可以查询只读视图，但不能通过汇总查询修改业务状态。

## 7. 模块通信

1. 当前请求必须立即获得结果时，调用对方 `public` 中的公开 Service。
2. 提交后执行的动作使用事务发件箱，例如目标激活后重评信号、持仓变化后取消高位通知。
3. RQ 消息只携带任务 ID、业务对象 ID 和冻结版本，不携带完整源码、密钥或大对象。
4. Redis 消息到达后，消费者必须回 PostgreSQL 读取正式状态。
5. 事件表示已经发生的事实，命令表示要求执行的动作，两者命名和处理分开。
6. 每个事件包含版本、事件 ID、发生时间、对象 ID、请求 ID、顺序键和去重键。
7. 跨模块锁定采用固定顺序：订阅 -> 目标 -> 信号 -> 持仓快照 -> 通知资格，避免并发死锁。

典型流程：

```text
行情模块保存有效行情
-> 发布“行情批次已完成”
-> 信号模块读取订阅、目标、行情和持仓公开快照
-> 信号模块只修改信号表并发布“信号已产生”
-> 通知模块读取事件和当前资格
-> 通知模块只修改通知表并执行渠道投递
```

## 8. 前端边界与顺序

前端公共区只包含按钮、输入、表格、弹窗、分页、确认、加载、空数据、部分成功、错误和版本冲突等基础能力。

策略编辑器归 strategies，行情图表归 market_data，回测图表归 backtests，不放入公共区。

每个业务页面按以下顺序完成：

1. 使用基础组件和模拟数据完成页面与交互状态。
2. 根据已冻结的 OpenAPI 契约生成类型。
3. 通过统一客户端连接真实接口。
4. 完成加载、空数据、部分成功、过期、超时、断网、后端不可用和 409 冲突状态。
5. 完成桌面端主流程和规定的手机查看能力。

主路由、导航和自动生成类型由批次整合负责人串行更新，业务子任务不并行修改这些公共文件。

## 9. 进程和容器

```text
postgres
redis
api
scheduler
dispatcher
watchdog
worker-realtime-data
worker-strategy
worker-notification
worker-bulk
sandbox-runner
frontend
```

逻辑队列可以细分，但当前服务器不为每个逻辑队列启动独立容器。实时数据、策略、通知和批量任务使用不同 Worker 组，防止全市场任务占用实时能力。

策略运行器必须满足：无网络、无数据库、无 Redis、无密钥、无宿主目录、只读根文件系统、非 root、资源限制、进程限制和硬超时。普通 API 和 Worker 不接触 Docker Socket。

## 10. 历史日线回填

按 5000 只股票、每年 250 个交易日估算：

- 10 年约 1250 万行。
- 20 年约 2500 万行。

当前服务器允许顺序回填，实施规则如下：

- 按股票或小批次流式获取和写入。
- 使用批量写入和按年份分区，不逐行提交。
- 每只股票完成后保存断点，重启后继续。
- 暂存数据提交后立即清理。
- 失败响应样本最多保留 7 天，正常原始响应不长期保存。
- 先回填 100 只股票，使用数据库真实占用推算全市场峰值，再放大全量任务。
- 回填任务支持暂停、继续、取消和仅重试失败项。

## 11. 施工批次

### 11.1 主流程：风险验证与公共底座

先验证 Provider、策略沙箱、任务故障恢复和回测耗时，再完成第 6.1 节公共底座与前端基础组件。公共底座通过验收并冻结后才启动业务子任务。

### 11.2 业务并行批次

| 批次 | 独立任务 |
|---|---|
| 1 | 交易日历；股票主数据；Provider 适配 |
| 2 | 明确时间调度；行情与日线；持仓 |
| 3 | 监控分组与订阅；策略编辑与隔离执行 |
| 4 | 四档目标；历史回填与数据修复；运维管理页面 |
| 5 | 信号状态机 |
| 6 | 业务通知联动；单股回测；仪表盘汇总 |
| 7 | 监控列表及全市场回测；策略发布、应用与回滚 |
| 最终 | 全系统串联、安全、故障恢复、性能和持续运行验证 |

不为凑满三个任务强行并行。目标先于信号，信号先于业务通知，单股回测先于策略发布。

## 12. 并行开发规则

- 每个子任务使用独立分支和独立工作区。
- 开始前冻结表归属、公开接口、事件、错误码和验收样例。
- 子任务只修改自己拥有的后端模块、前端 feature 和测试。
- 公共底座、Compose、迁移环境、主路由和生成接口类型由主流程维护。
- 需要变更公共契约时，子任务停止并提交变更说明，由主流程统一处理。
- 每批完成后逐个审查和合并，再运行迁移、全量测试和容器验收。
- 本批未通过，下一批不启动。

## 13. 测试与质量门槛

测试分为：

1. 纯规则测试：边界、滞回、配置继承、重试分类和回测指标。
2. 模块契约测试：Provider 样本、策略输入输出、通知假服务和公开接口。
3. PostgreSQL 集成测试：唯一约束、锁、不可变历史、发件箱和迁移。
4. Docker 进程测试：API、分发器、看门狗和 Worker 的真实跨进程流程。
5. 端到端测试：先一只股票，再二十只股票，覆盖目标、行情、信号和通知。
6. 故障测试：断 Redis、杀 Worker、重复消息、时钟偏差、网络延迟和恶意策略。
7. 性能测试：日线批次、单股回测和批量任务不影响实时与通知。

每个模块的退出条件：

- 正常、空数据、部分成功、重复、超时和重启场景通过。
- 告警和诊断信息可见。
- 数据库迁移可从空库执行。
- 模块测试通过，完整测试无回归。
- 敏感信息不进入返回、日志、审计差异或任务载荷。

## 14. 首个可用闭环

公共底座完成后，首个业务闭环的验收路径是：

```text
管理员登录
-> 导入交易日历和一只股票
-> 建立监控订阅和手工四档目标
-> 手工获取一批有效行情
-> 产生或静默建立信号状态
-> 形成网页通知记录
-> 查询任务、日志和审计
```

企业微信、邮件、策略目标和批量回测在各自依赖完成后逐步接入，但不得破坏此基础闭环。
