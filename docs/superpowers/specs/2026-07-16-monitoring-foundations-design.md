# 阶段 3 监控基础设计

## 1. 目标与范围

本设计落实当前 V3.1 基线的阶段 3 第一批，为后续手工目标、信号状态机和通知业务接入提供稳定基础。

本批包含：

- 监控分组及分组成员；
- 监控调度定义及不可变修订；
- 当前持仓及不可变持仓历史；
- 监控订阅、订阅修订和调度执行记录；
- 上述模块的认证、审计、事务发件箱、公开 API 和恢复语义。

本批不包含：

- 四档目标内容、目标计算和目标复核；
- 信号状态机、滞回和正式信号通知；
- 策略编辑、策略发布和沙箱；
- 企业微信或邮件真实发送；
- 持仓数量、成本、成交、盈亏和历史生效时间回填。

## 2. 施工边界与依赖顺序

公共契约、错误码、事件名称、数据库迁移主链和主路由由主流程串行维护。公共契约验收后，第一批最多并行三个互不依赖的模块：

1. `watchlists`：只拥有分组和分组成员；
2. `monitor_schedules`：只拥有调度定义和不可变修订；
3. `positions`：只拥有当前持仓和不可变历史。

三者通过公开 Application、Service、冻结契约和事务发件箱协作，不引用其他模块的 ORM、仓储或表。三项接入后，主流程串行实现 `monitor_subscriptions` 和 `schedule_occurrence`，避免订阅在并行期提前依赖尚未稳定的调度、目标或策略契约。

## 3. 监控分组

### 3.1 数据归属

- `watchlist`：管理员拥有的分组，保存名称、说明、显示顺序、版本、归档状态和时间；
- `watchlist_item`：分组与股票关系，保存创建时间和来源；同一分组不能重复包含同一股票。

分组只负责组织和筛选，不代表正式监控。股票加入多个分组不能产生重复行情、信号、任务或通知。

### 3.2 行为

- 创建、改名、排序和归档使用乐观锁；
- 删除分组等价于归档，不物理删除历史；
- 归档分组不自动暂停订阅；
- 添加成员通过股票模块公开身份接口验证正式 A 股范围；
- 单项重复添加返回已有关系，不创建重复记录；
- 批量添加逐项返回 `CREATED/REUSED/REJECTED/FAILED`，一项失败不回滚其他项；
- 从一个分组移除不影响其他分组；
- 移除股票最后一个分组关系时只返回 `pause_recommended=true`，由用户显式确认是否暂停订阅，分组模块不得直接修改订阅。

### 3.3 事件

成功修改后写入 `watchlist.updated`。事件只包含分组、动作、成员股票、版本和请求原因，不包含其他模块内部记录。

## 4. 监控调度

### 4.1 数据归属

- `monitor_schedule`：当前调度指针、名称、当前版本、版本号、归档状态和乐观锁版本；
- `monitor_schedule_revision`：不可变时间列表、时区、原因、创建者、请求编号、内容哈希和创建时间。

调度统一使用 `Asia/Shanghai` 解释页面时间，正式计划时刻转换为 UTC 保存和执行。

### 4.2 时间规则

- 精确到分钟；
- 最多 20 个；
- 自动升序；
- 不允许重复；
- 只允许 `09:30–11:30` 和 `13:00–15:00`；
- 空列表可以保存为未配置调度，但不能用于启用订阅；
- 修改产生新修订，不覆盖旧版本；
- 恢复历史版本通过复制内容生成新版本，不能把当前指针直接倒退到旧记录。

同一内容重放返回当前版本；同一幂等键提交不同内容返回冲突。已有行情批次继续使用冻结调度版本，新版本只影响尚未创建的批次。

### 4.3 事件

调度创建、修改、恢复和归档写入 `monitor_schedule.changed`，包含调度编号、当前修订、版本号、时间列表和原因。

## 5. 持仓

### 5.1 数据归属

- `user_position`：每只股票最多一条当前记录，保存 `HOLDING/NOT_HOLDING`、版本、修改时间、来源和最新历史编号；
- `user_position_history`：不可变保存前后状态、生效时间、备注、来源、请求编号、幂等键和持仓版本。

未创建当前记录时公开查询返回默认 `NOT_HOLDING`，但不因此写入数据库。备注去除首尾空白后最多 500 字，不允许回填过去生效时间。

### 5.2 状态变化

- 相同状态重复提交返回 `POSITION_UNCHANGED`，不新增历史和业务事件，只保留轻量审计；
- 真实变化在一个事务中锁定当前记录、插入历史、更新当前状态、写高风险审计和事务发件箱；
- 每次真实变化都写 `position.changed`；
- `NOT_HOLDING -> HOLDING` 另写 `position.became_holding` 和 `position.high_review_requested`；
- `HOLDING -> NOT_HOLDING` 另写 `position.became_not_holding` 和 `high_notification_cancel_requested`；
- 本模块不直接读取或修改目标、信号和通知表；后续模块消费事件并重新检查最新持仓版本；
- 持仓可以独立于监控订阅存在，退市或停牌不阻止用户更新持仓事实。

事件幂等键包含股票、持仓版本和事件类型。高位复核失败不得回滚已经提交的持仓变化。

## 6. 监控订阅

### 6.1 数据归属

- `monitor_subscription`：每只股票最多一条未归档订阅，保存生命周期、当前修订、配置版本和时间；
- `monitor_subscription_revision`：不可变保存调度及其冻结版本、目标模式、策略版本、参数快照、滞回、通知策略、原因和内容哈希。

生命周期使用：

- `CONFIGURING`：订阅存在但尚未满足启用条件；
- `ENABLED`：参与未来计划批次；
- `PAUSED`：立即停止未来批次和正式信号资格；
- `ARCHIVED`：默认隐藏，不参与业务。

归档前必须先暂停。恢复归档订阅复用原订阅编号并进入 `PAUSED`，不能静默重新启用。

### 6.2 创建与配置

创建时通过股票公开身份契约验证 A 股范围、上市状态和主数据版本。新订阅允许调度、目标或策略缺失，并进入 `CONFIGURING`。同一股票重复创建返回现有未归档订阅；不同内容使用同一幂等键返回 409。

配置修改产生不可变修订。调度引用必须同时保存调度编号和修订编号。已开始的行情批次继续使用原冻结修订，后续批次读取新版本。

阶段 3 第一批只实现 `MANUAL/STRATEGY` 模式和可空引用的结构，不伪造目标或策略记录。目标和策略模块接入前，启用验证必须因目标不可用返回稳定错误，不能绕过门槛。

### 6.3 暂停与并发安全

暂停在事务提交后立即生效：

- 未来调度扫描不再包含；
- 已排队但未领取的订阅项由后续取消流程处理；
- 已取得行情可以保存；
- 信号提交前必须读取当前订阅状态和配置版本；
- 旧任务只能记录 `SUPERSEDED`，不能产生新信号或通知。

所有修改使用 `expected_version`。版本不符返回 `MONITOR_SUBSCRIPTION_VERSION_CONFLICT`。

## 7. 调度执行记录

`schedule_occurrence` 保存调度类型、调度定义、计划 UTC 时间、冻结订阅集合、状态、领取时间、任务编号、错误码和时间。唯一键为调度类型、调度定义和计划 UTC 时间。

调度器每 10 秒扫描一次当前交易日内的到期时刻，只领取 60 秒宽限期内的记录；超过宽限期标记 `MISSED`，停机恢复不补跑盘中批次。同一计划时刻到期的启用订阅合并成一个预期股票集合，并创建独立实时行情任务。前一批次未完成时，新时刻不能复用旧行情，超过自身宽限期仍标记 `MISSED`。

调度执行通过公开日历窗口、订阅快照和任务 Service 协作，不直接访问其内部表。创建执行记录、任务和 `schedule_occurrence.created/missed` 发件箱事件必须保持事务一致。

## 8. API 与权限

公开接口按 V3.1 基线提供：

- `/api/v1/watchlists` 及成员单项、批量接口；
- `/api/v1/monitor-schedules`、版本列表和恢复接口；
- `/api/v1/positions`、单股、历史、持仓、清仓和批量接口；
- `/api/v1/monitor-subscriptions` 及启用、暂停、归档、恢复接口；
- `/api/v1/schedule-occurrences` 查询接口。

所有读取要求登录。创建、修改、归档、恢复、持仓变化和订阅启停要求已验证写身份、可信 Origin、CSRF、原因和幂等键。批量接口逐项返回结果，不因单项业务失败回滚其他项。

本批只预留 `check-now/diagnose` 的公开命令契约；正式行为在目标和信号模块接入时实现，未接入前返回稳定的能力不可用错误，不能伪造正式结果。

## 9. 错误处理

稳定错误至少包含：

- `WATCHLIST_NOT_FOUND`、`WATCHLIST_VERSION_CONFLICT`、`WATCHLIST_ARCHIVED`；
- `WATCHLIST_ITEM_REJECTED`、`WATCHLIST_BATCH_PARTIAL`；
- `MONITOR_SCHEDULE_NOT_FOUND`、`MONITOR_SCHEDULE_INVALID`、`MONITOR_SCHEDULE_VERSION_CONFLICT`；
- `POSITION_SYMBOL_INVALID`、`POSITION_VERSION_CONFLICT`、`POSITION_UNCHANGED`；
- `MONITOR_SUBSCRIPTION_NOT_FOUND`、`MONITOR_SUBSCRIPTION_CONFLICT`、`MONITOR_SUBSCRIPTION_VERSION_CONFLICT`；
- `MONITOR_SUBSCRIPTION_NOT_READY`、`MONITOR_SUBSCRIPTION_ARCHIVED`；
- `SCHEDULE_OCCURRENCE_MISSED`、`MONITOR_BACKEND_UNAVAILABLE`。

数据库超时或不可用映射为 503。权限或非法输入在事务开始前拒绝。事务发件箱失败时业务修改必须整体回滚；Redis 或分发失败不回滚已提交的正式数据。

## 10. 验收与测试

每个模块至少覆盖：

- 正常创建、读取、修改、归档和恢复；
- 空列表、空分组、默认未持仓和配置不完整订阅；
- 同内容重放、同键异内容、重复成员和重复状态；
- 非法证券、退市订阅、非法时间、超过 20 个时间和重复时间；
- 乐观锁冲突、同股并发、批量部分失败和数据库唯一约束；
- 审计或发件箱失败时整笔回滚；
- 进程重启后的调度领取恢复、过期不补跑和重复扫描去重；
- 暂停与行情批次并发时的最终资格门槛；
- 迁移升级、回退、再升级和应用角色权限；
- 认证、Origin、CSRF、必需幂等头、具体 OpenAPI 响应类型；
- PostgreSQL、Redis、API、调度器和相关 Worker 的容器验收。

完成标准不是代码存在，而是三个并行模块分别通过定向验收，主流程逐个接入后通过全量测试和容器验收。
