# 监控股前复权刷新设计

## 1. 目标与范围

本模块完成 V3.1 阶段 2 的监控股前复权能力：按冻结窗口从东方财富获取前复权日线，建立不可变数据集，完整校验后原子切换当前有效版本；失败时继续使用旧数据并标记过期。

本阶段不创建监控订阅、策略、目标或信号。刷新任务显式冻结股票、日期窗口、触发原因和输入版本；阶段 3 的监控模块负责决定哪些股票允许自动刷新，阶段 4 的策略模块负责提供策略窗口并消费完成事件。

全市场不复权日线继续永久保存。全市场回测前复权数据仍按股票临时获取和释放，不写入本模块的长期数据集。

## 2. 方案选择

采用“不可变数据集头 + 规范化日线明细 + 刷新运行记录”。不采用 JSON 大字段，因为数据库无法可靠约束日期唯一性和价格结构；不采用覆盖式更新，因为刷新失败或进程中断时难以证明旧数据完整可用。

V3.1 列出的核心业务表仍为 `qfq_dataset` 和 `qfq_refresh_run`。新增 `qfq_dataset_bar` 作为数据集的物理明细表，不形成新的业务聚合；它用于保存可查询、可约束的 OHLCV 明细。

## 3. 模块边界

新建独立 `qfq` 模块，拥有：

- 前复权数据集及其明细；
- 刷新运行记录；
- 窗口、数据质量和校验和规则；
- 当前版本切换事务；
- 查询与人工刷新接口；
- `qfq_refresh.completed` 和 `qfq_refresh.failed` 事件。

模块只通过公开契约调用：

- 股票查询契约：解析 `symbol`、`security_id` 和股票状态；
- 日线读取契约：确认目标日存在有效不复权日线及其版本；
- Provider 契约：调用东方财富 `HISTORICAL_DAILY_QFQ`；
- 任务服务：提交、执行和恢复 `QFQ_REFRESH`；
- 事务发件箱：提交完成或失败事件。

禁止引用股票、日线、Provider 或未来监控模块的内部模型和数据访问层。

## 4. 数据模型

### 4.1 `qfq_dataset`

每行表示一套不可变数据集，至少保存：

- `id`、`security_id`、`symbol`、单股递增 `version`；
- 请求窗口和实际窗口；
- `as_of_date`、数据源、Provider 能力版本；
- 复权基准日期和基准收盘价；
- 行数、内容校验和；
- 生命周期 `STAGING/CURRENT/SUPERSEDED`；
- 新鲜度 `FRESH/STALE` 及过期原因；
- 创建时间、激活时间和替代时间。

同一股票最多一套 `CURRENT` 数据集。生命周期和新鲜度分离，保证旧数据可以同时是“仍在使用”与“已经过期”。

### 4.2 `qfq_dataset_bar`

以 `(dataset_id, trade_date)` 为主键，保存 `open/high/low/close/volume/amount`。数据库约束价格为正、OHLC 关系有效、数量非负；删除未激活或已清理的数据集时级联删除明细。

### 4.3 `qfq_refresh_run`

保存任务、股票、冻结窗口、触发原因、输入日线版本、来源、阶段、行数、校验和、候选和激活数据集、错误码、重试信息及时间。状态为：

```text
PENDING -> FETCHING -> VALIDATING -> COMMITTING -> SUCCEEDED
                    \-> FAILED
                    \-> TIMED_OUT
                    \-> SUPERSEDED
```

同一股票同一冻结输入使用稳定内容哈希去重。相同幂等键不同内容返回冲突；迟到旧任务不能覆盖更新的数据集。

## 5. 刷新流程与事务

1. API 或内部调用提交 `QFQ_REFRESH`，冻结股票、日期窗口、当前交易日历给出的预期交易日列表、目标日、不复权日线版本、触发原因和请求身份。
2. 独立 `qfq-refresh` Worker 使用东方财富前复权能力抓取数据，网络调用不占用数据库事务。
3. 校验日期范围、升序、重复、行数、目标日、OHLC、数量、来源和字段完整性；生成规范化内容校验和。
4. 校验失败时记录运行失败；若存在当前数据集，将其新鲜度改为 `STALE`，但不修改其明细和当前身份。
5. 校验成功后在单一数据库事务中：锁定该股票当前数据集，核对冻结输入仍有效，写候选数据集和明细，将旧版本改为 `SUPERSEDED`，将候选改为 `CURRENT/FRESH`，完成运行记录并写完成事件发件箱。
6. 任何事务错误全部回滚，旧数据集仍为当前版本。Redis 或分发失败不回滚正式数据，事件由发件箱恢复。

当前完整数据集和上一套完整数据集保留明细。更早数据集保留元数据，明细由维护任务清理。清理不得删除当前、上一套、仍被运行记录引用或仍在事务发件箱中的数据集。

## 6. 数据门槛

刷新窗口不使用全局固定 `1000`。任务必须明确提供起止日期或由未来策略版本换算后的冻结窗口。

正式激活至少要求：

- 股票和请求窗口有效；
- 目标日存在已提交的不复权日线；
- Provider 返回目标日及请求所需的完整可用窗口；
- 最后一日前复权收盘与同日不复权收盘满足复权基准一致性；
- 实际日期列表与冻结的预期交易日列表完全一致，没有缺失、重复或倒序；
- 价格和金额符合正式数据库精度及数量级，没有非法 OHLC、负成交量或非有限数值；
- 刷新期间股票、输入日线和当前数据版本没有失效。

合理停牌或非交易日由交易日历和日线契约解释，不能仅因 Provider 无返回自行推断。

## 7. 公开接口

### `GET /api/v1/qfq-data/{symbol}`

需要登录。支持日期和分页参数，返回当前数据集元数据、新鲜度、窗口、校验和和日线明细。没有当前数据集时返回 `QFQ_DATA_NOT_FOUND`。

### `POST /api/v1/qfq-data/{symbol}/refresh`

需要登录、可信来源、CSRF、确认、原因和 `Idempotency-Key`。请求包含明确的起止日期和目标日，返回 `202/JOB_ACCEPTED/job_id`，API 不同步等待 Provider。

## 8. 内部事件与未来接入

- `qfq_refresh.completed`：包含股票、数据集、版本、窗口、行数、校验和、输入日线版本和原因。
- `qfq_refresh.failed`：包含股票、窗口、稳定错误码、是否仍有旧数据、旧数据是否过期和重试建议。

阶段 3 的监控订阅通过公开 Service 提交自动刷新；阶段 4 的目标服务消费完成事件并再次检查订阅、策略和数据版本。前复权模块不直接调用未来模块。

## 9. 错误码

- `QFQ_DATA_NOT_FOUND`
- `QFQ_WINDOW_INVALID`
- `QFQ_DAILY_GATE_NOT_MET`
- `QFQ_PROVIDER_FAILED`
- `QFQ_EMPTY_RESULT`
- `QFQ_VALIDATION_FAILED`
- `QFQ_REFRESH_CONFLICT`
- `QFQ_INPUT_SUPERSEDED`
- `QFQ_REFRESH_TIMED_OUT`
- `QFQ_BACKEND_UNAVAILABLE`

错误只返回稳定代码和脱敏摘要，不保存完整第三方响应。

## 10. 任务和部署

新增逻辑队列 `qfq-refresh` 和独立 Worker。单股刷新建议软超时 240 秒、硬超时 300 秒，符合 V3.1 的单股前复权 5 分钟上限。Worker 重启后依据运行记录和任务栅栏恢复；旧栅栏令牌不得提交结果。

## 11. 验收标准

至少覆盖：

- 正常刷新、查询和版本切换；
- 空结果、重复日期、倒序、窗口不足、目标日缺失和非法 OHLC；
- Provider 超时、失败、熔断和无自动备用来源；
- 相同请求幂等重放与不同内容冲突；
- 同股并发刷新、迟到旧任务和栅栏失效；
- 失败时旧数据保持当前且标记过期；
- 成功事务中数据集、明细、运行记录和发件箱原子提交；
- Redis 失败后的发件箱恢复；
- Worker 中断和恢复；
- 正常、失败和回滚迁移；
- API 权限、非法输入、分页和明确响应类型；
- `qfq-refresh` Worker 只监听自己的队列，不影响实时和日线任务。
